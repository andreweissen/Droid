Class: Client

client~Client(token, config, lang)

The Client class serves as the beating heart of the application of the same name. It is responsible for coordinating the behavior of the bot, loading commands, evaluating each new message posted to server channels, and replying to user queries as required.

As the central class of the bot application, Client is the only class that may interact with other users in the server channels through the posting of messages. In previous versions of the application, various command subclasses extending Command would likewise post messages in the server. However, this spaghetti approach was eventually refactored and replaced with the passing of each subclass a reference to addReply bound to the Client class instance to ensure all message logging remained during the exclusive purview of the central Client class.

Methods

addReply(message, langMessage, deleteMessagesopt) → {void}

The addReply method is used to log a new reply directed at the user invoking the given command that provides information about the state of the request. A copy of this function bound to the Client class instance to which the method belongs is passed to each command subclasses's execute method to ensure that the Client class is the only module to interact with the server channels. Subclasses extending Command should only contain application logic related to those commands, leaving the posting of results to this method.

onMessage(message) → {void}

Arguably the most important method of the Client class, onMessage serves as the primary event listener callback handling "message" events. As such, it is responsible for checking to see if the bot application should take an interest in the most recent message, namely the Discord.Message instance that is passed as the method's sole parameter. If the message is determined to constitute a command invocation, the method will check to see if a command matching the invoked name exists in Client#commands, the Discord.Collection containing all previously instantiated command subclasses corresponding to available command functionality. If so, the method invokes the relevant implementation of Command#execute to pass the execution progression off to the relevant subclass.

On a more detailed level, the method begins by checking if the user has attempted to post a non-verification message in the verify channel, logging the message in the moderator logs channel if so before deleting the message and adding a timed reply in response. Otherwise, if the message was a normal user post in another channel rather than a command invocation, the method ignores it and returns to await the next message.

However, if the message does constitute a wellformed command invocation, the method will validate the input and attempt to locate an available command matching the desired command. If one is found, the method will then query Client#commands to see if an instance of the appropriate command subclass extending Command exists, invoking Client#loadCommand if not. Execution is then handed off to the Command#execute method to address the required command action.
